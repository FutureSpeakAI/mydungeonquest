---
name: GitHub push refusals through Replit lanes
description: Diagnosing PUSH_REJECTED from the gitPush callback / Git pane when GitHub's real refusal text is hidden
---
- The gitPush callback and the user's Git pane both flatten EVERY GitHub refusal (OAuth scope, secret push-protection, size, non-FF, org policy) into `PUSH_REJECTED` / "Unknown Git Error". Never build theories from the wrapper text.
- **Why:** GitHub's stderr prose ("refusing to allow an OAuth App to create or update workflow `.github/workflows/x.yml` without `workflow` scope", GH009 secret lists) is swallowed by both lanes; a long session burned rounds on wrong theories (org write rights, size ceilings, secret scanning) that the real message would have answered instantly.
- **How to diagnose:** (1) 27-byte orphan-branch probe proves auth/app-access/write. (2) Deliver growing prefixes of history: `git update-ref` a scratch branch + `git symbolic-ref HEAD refs/heads/<scratch>` (no checkout, worktree untouched — the callback pushes the current branch). (3) Bisect the first failing range; every PASS advances the remote so deltas keep shrinking. (4) When refusal pins to one commit, read its FILE LIST before its content — refusal classes are often path-based.
- **Proven refusal class:** any push that adds or modifies `.github/workflows/*` fails without the `workflow` OAuth scope — every branch, every repo, any size; Replit's lanes lack that scope. Fix: strip workflow files from history, keep them on disk gitignored, re-add via GitHub web UI (or a workflow-scoped PAT from outside).
- A committed `private_key.pem` was also scrubbed (right call for security) but its refusal was never separately proven — the workflow file poisoned the same range and masked all other evidence. One range can hide MULTIPLE refusal classes; cure-verify after each scrub.
- Size was NEVER a proven refusal cause (11.8MB passed, 10.0MB failed — both verdicts tracked the poisoned commit, not bytes). Don't infer transport ceilings from failures that share a poisoned commit; ~10MB delta hops remain a decent delivery ladder when unsure.
- First push to a "new" repo refused? `git ls-remote` it anonymously first: repos are often non-empty (auto-init README, or a forgotten old snapshot of the same project). If public, cure without credentials or force: anonymous `git fetch <url> main` + `git merge -s ours --allow-unrelated-histories` — tree stays byte-identical, remote tip enters ancestry, the push turns fast-forward.
- After any history scrub, audit tags before ANY tag push: tag pointers unreachable from the rewritten branch pin the OLD line, and pushing them resurrects everything scrubbed (bulk assets, keys). Retarget by matching commit subjects on the rewritten branch first.
